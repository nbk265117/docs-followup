1. VUE D'ENSEMBLE DU PROJET

▼

1.1 Contexte

Le projet BOA Follow Up vise à digitaliser et moderniser le processus de suivi et de pilotage des pratiques commerciales et managériales du réseau Bank of Africa. Cette solution remplacera l'outil Appsheet actuel par une architecture Microfrontend (5 projets Angular 20 + 1 Backend) moderne, performantes et sécurisées.

1.2 Architecture Applicative

Architecture Microfrontend à 6 Applications :

1 Backend API : Spring Boot (API REST commune)
  → msiboa-oc-api-followup-serveur (GitLab)
  → Recette : https://www.followup-api.bmcebank.co.ma

5 Projets Frontend (Microfrontend Single-SPA) :
  • Root Config (Shell) : Orchestrateur Single-SPA + Keycloak Init
    → msiboa-cc-mf-followup-root (port 9000)

  • Layout : Sidebar + Header partagés
    → msiboa-cc-mf-followup-layout (port 9099)

  • Follow Up (Client) : Application terrain pour les évaluations
    → msiboa-cc-mf-followup-client (port 9080)
    → Recette : https://www.followup.bmcebank.co.ma

  • Backoffice (Admin) : Application admin pour la gestion
    → msiboa-cc-mf-followup-admin (port 9080)
    → Recette : https://www.followup-admin.bmcebank.co.ma

  • Utility (Shared) : Ressources partagées (session$, i18n, images)
    → msiboa-cc-mf-followup-utility (port 8080)

1.3 Objectifs Stratégiques

  • Digitaliser le processus d'évaluation des pratiques commerciales et managériales
  • Fournir une interface moderne et intuitive accessible sur tous supports (desktop, mobile, tablette)
  • Permettre la saisie d'observations terrain avec cache navigateur
  • Assurer la traçabilité complète de toutes les actions via des audit logs détaillés
  • S'intégrer avec le système d'information BOA (SSO Keycloak, ASAL pour annuaire collaborateurs)
  • Garantir un haut niveau de sécurité conforme aux standards bancaires
  • Faciliter le pilotage et l'analyse des données par le management

1.4 Périmètre Fonctionnel par Application

📱 Application Follow Up (Terrain)

  • Évaluations : Saisie observations, soumission, consultation historique
  • Formulaires : Consultation des modèles d'évaluation assignés
  • Dashboard : Vue d'ensemble des évaluations (personnelles, équipe, région)
  • Tableaux de Bord & Analytics : KPIs, statistiques, analytics (selon rôle : Directeurs, DG)
  • Notifications : Alertes et rappels
  • Profil : Gestion du profil utilisateur

🔧 Application Backoffice (Administration)

  • Gestion Utilisateurs : CRUD utilisateurs, attribution des rôles
  • Gestion Habilitations : Configuration RBAC, permissions
  • Gestion Modèles d'Évaluation : CRUD formulaires, champs dynamiques
  • Structure Organisationnelle : Gestion agences, régions, zones
  • Audit Logs : Consultation traçabilité, rapports
  • Configuration Système : Paramètres globaux

2. ARCHITECTURE GLOBALE

▼

L'architecture proposée est une architecture Microfrontend (Single-SPA 5.9) avec 5 projets frontend partageant le même backend API. Cette architecture permet :

  • Une expérience utilisateur optimisée pour chaque type d'utilisateur
  • Un déploiement indépendant de chaque microfrontend
  • Un Shell Container (Root Config) qui orchestre le chargement des MFEs
  • Un Layout partagé (Sidebar, Header) entre toutes les applications
  • Des ressources partagées via le module Utility (i18n, session$, images)
  • Une communication inter-MFE via Props, Custom Events et RxJS
  • Une authentification centralisée via Keycloak 26.1

13. PERFORMANCE & OPTIMISATIONS

▼

13.1 Optimisations Frontend Angular

Lazy Loading Modules

Techniques : Angular Router avec loadChildren pour le chargement différé des feature modules
• Réduction du bundle initial
• Chargement à la demande des modules

Change Detection Strategy

OnPush Strategy : changeDetection: ChangeDetectionStrategy.OnPush
• Réduit les cycles de détection de changements
• Améliore significativement les performances
• Utiliser avec Observables (async pipe)

Virtual Scrolling

CDK Virtual Scrolling : @angular/cdk/scrolling
• cdk-virtual-scroll-viewport pour listes longues
• Rend uniquement les éléments visibles
• Intégré au Angular CDK

TrackBy Functions

Optimisation *ngFor : Utiliser trackBy pour éviter les re-renders inutiles des listes
• *ngFor="let item of items; trackBy: trackById"

AOT Compilation & Tree Shaking

Build optimisé : Ahead-of-Time compilation activée par défaut en production
• Tree shaking pour éliminer le code mort
• Minification et compression automatiques

13.2 Optimisations Backend

Pagination & Filtering

Spring Data JPA : Pagination native avec Pageable + filtres dynamiques

Database Query Optimization

  • N+1 Query Problem: Utiliser @EntityGraph ou JOIN FETCH
  • Index strategy: Index sur colonnes de filtrage fréquentes
  • Projection: Utiliser DTO projections au lieu d'entités complètes
  • Connection Pooling: HikariCP (par défaut Spring Boot)
